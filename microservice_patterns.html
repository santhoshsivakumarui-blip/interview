A single front door for every client request. The gateway routes to the right internal service, so clients never talk to services directly — and never know how many there are, or where they live.
traffic flow
why it exists
  • +One entry point — clients don't hardcode 8 different service URLs
  • +Cross-cutting concerns live in one place: auth, rate limiting, request logging, CORS
  • +Services can be added, split, or moved without touching client code
  • −Becomes a single point of failure if not made highly available
  • −Risk of turning into a "God object" if business logic creeps in
express-style route (your stack)
const proxy = require('express-http-proxy');

app.use('/api/orders', authenticate,
  proxy('http://orders-svc:4001'));

app.use('/api/payments', authenticate,
  proxy('http://payments-svc:4002'));
// gateway owns auth + routing, nothing else
Instances scale up and down constantly — you can't hardcode IPs. Each instance registers itself with a registry; clients (or the gateway) look up a live address before every call.
registration + lookup

Grey pulses = heartbeats (instances proving they're alive). Orange = client asking the registry "who's healthy?" Teal = the actual traffic, going straight to the chosen instance — the registry is never in the request path itself.

why it exists
  • +Instances can scale/die/restart freely — nothing is hardcoded
  • +Enables client-side load balancing (pick any healthy instance)
  • +Unhealthy instances get evicted automatically when heartbeats stop
  • −Registry itself needs high availability — it's now critical infrastructure
  • −Stale registry data can route traffic to a dead instance briefly
in practice
// K8s does this for you via Services + kube-proxy
// DNS-based discovery, no registry code needed:
fetch('http://orders-svc.default.svc.cluster.local')

// or explicit, self-hosted:
consul.register({name:'orders', address, port, checkTTL:'10s'});
When a downstream service is failing, hammering it with retries makes things worse. A circuit breaker trips open after enough failures, fails fast with a fallback, then tests the service again before fully trusting it.
state machine — cycling automatically CLOSED OPEN HALF-OPEN

Closed: requests pass straight through to the service.

the three states
  • 1Closed — normal traffic flows; breaker counts failures
  • 2Open — failure threshold hit; calls short-circuit instantly to a fallback, no network call made
  • 3Half-Open — after a cooldown, one test request is allowed through to check recovery
opossum (node.js)
const CircuitBreaker = require('opossum');

const breaker = new CircuitBreaker(callPaymentsApi, {
  timeout: 3000,
  errorThresholdPercentage: 50,
  resetTimeout: 10000
});
breaker.fallback(() => ({status: 'queued_for_retry'}));
No distributed transactions across services — each step commits locally. If a later step fails, compensating actions undo the earlier steps in reverse. This is orchestration-style: a coordinator drives the sequence.

orchestrated saga — order checkout

Press Play to run the checkout saga — including what happens when the last step fails.

why it exists
  • +No two-phase commit across services — each keeps full local autonomy
  • +Works across different databases/tech stacks per service
  • −Compensating logic must be written for every step — real extra code
  • −No true isolation — a client can briefly see a partially-completed order
orchestration vs choreography
Orchestration (shown above): one coordinator explicitly calls each service in order — easy to follow, but the coordinator is a central dependency.

Choreography: no coordinator — each service publishes an event, the next service reacts to it. More decoupled, harder to trace end-to-end.
The publisher doesn't know or care who's listening. It emits an event to a broker; any number of subscribers consume it independently, at their own pace. Add a new subscriber without touching the publisher at all.
order placed → fan-out

Notice the fan-out isn't simultaneous — each subscriber pulls the event independently. One slow consumer never blocks the others, or the publisher.

why it exists
  • +Publisher and subscribers are fully decoupled — deploy independently
  • +New consumers can subscribe later without any change upstream
  • +Broker buffers events, so a temporarily-down subscriber doesn't lose data
  • −Eventual consistency — subscribers see the event slightly after the fact
  • −Harder to trace a request across an event chain than a direct call
kafkajs example
// publisher
await producer.send({
  topic: 'order.placed',
  messages: [{ value: JSON.stringify(order) }]
});

// subscriber (independent service)
await consumer.subscribe({ topic: 'order.placed' });
await consumer.run({ eachMessage: sendConfirmationEmail });
Each service owns its data exclusively — no other service touches its database directly, ever. Cross-service data needs go through APIs or events, never a shared join.
ownership boundary
why it exists
  • +Each service can change its schema without coordinating a migration across teams
  • +Different services can use different database technology where it fits (SQL vs document vs cache)
  • +No shared-table lock contention across unrelated services
  • −No cross-service joins — you need API composition or a read-model instead
  • −Data consistency across services becomes eventual, not immediate
related: CQRS
When you need to query across services often, don't join their databases — build a separate read model (a denormalized query-side database) kept in sync via the events each service already publishes. Writes go to each service's own DB; reads hit the purpose-built read model. This is the natural next pattern once database-per-service starts making reporting painful.
How these compose in a real system — a client hits the API Gateway, which uses Service Discovery to find a healthy instance, wraps the call in a Circuit Breaker, and if the operation spans services, it's coordinated as a Saga. Side effects (emails, analytics) go out via Pub/Sub, and every service guards its own data behind Database-per-Service. They're not competitors — they're a stack.
