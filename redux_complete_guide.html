Redux core — the foundation

Redux is a predictable state container. Everything flows in one direction through three fixed roles.

UI dispatches action
→
Reducer computes new state
→
Store notifies subscribers
→
UI re-renders
Store
Single object that holds all application state. Created once with configureStore().
Action
Plain object describing what happened. Must have a type field. Optionally carries a payload.
Reducer
Pure function: (state, action) → newState. No side effects, no mutations.
Dispatch
The only way to trigger a state change. Sends an action to the store.
Selector
Function that reads from state. Use useSelector() in components. Memoize with createSelector().
Slice
RTK pattern — bundles reducer + actions for one feature. Created with createSlice().
// 1. Create a slice (Redux Toolkit — the modern way) import { createSlice } from '@reduxjs/toolkit' const counterSlice = createSlice({ name: 'counter', initialState: { value: 0 }, reducers: { increment(state) { state.value += 1 }, // Immer: mutate freely decrement(state) { state.value -= 1 }, addAmount(state, action) { state.value += action.payload }, }, }) export const { increment, decrement, addAmount } = counterSlice.actions export default counterSlice.reducer
// 2. Configure store import { configureStore } from '@reduxjs/toolkit' import counterReducer from './counterSlice' const store = configureStore({ reducer: { counter: counterReducer }, }) // 3. Use in a component import { useSelector, useDispatch } from 'react-redux' import { increment, addAmount } from './counterSlice' function Counter() { const value = useSelector(state => state.counter.value) const dispatch = useDispatch() return <button onClick={() => dispatch(increment())}>{value}</button> }
1
Install
npm i @reduxjs/toolkit react-redux — RTK is the official recommended way. Never install plain redux alone anymore.
2
Understand Immer
RTK uses Immer under the hood. Inside createSlice reducers you can appear to mutate state — Immer converts it to an immutable update behind the scenes.
3
Selectors are your read API
Components should never know the shape of the store. Put selector logic in the slice file. Use createSelector from RTK when deriving data to avoid re-renders.
4
Normalise relational data
Use createEntityAdapter() for collections (users, posts, events). It gives you CRUD reducers and selectors for free.

Redux Thunk — async actions

A thunk is a function that returns another function instead of an action object. The middleware calls it with dispatch and getState, letting you do async work and dispatch actions when you're done.

Mental model: Normally dispatch(action) expects a plain object. Thunk middleware intercepts functions: if you dispatch a function, it calls it with (dispatch, getState) instead of sending it to the reducer.
dispatch(thunkFn)
→
Middleware intercepts
→
thunkFn(dispatch, getState)
→
dispatch real actions
// Manual thunk (plain function returning a function) export const fetchUser = (userId) => async (dispatch, getState) => { dispatch({ type: 'users/fetchStart' }) try { const user = await api.getUser(userId) dispatch({ type: 'users/fetchSuccess', payload: user }) } catch (err) { dispatch({ type: 'users/fetchError', payload: err.message }) } } // Usage in component dispatch(fetchUser('user-123'))
// RTK's createAsyncThunk — removes all the boilerplate above import { createAsyncThunk, createSlice } from '@reduxjs/toolkit' export const fetchUser = createAsyncThunk( 'users/fetchById', // action type prefix async (userId, thunkAPI) => { const response = await api.getUser(userId) return response.data // becomes action.payload } ) // Auto-generates: users/fetchById/pending|fulfilled|rejected const usersSlice = createSlice({ name: 'users', initialState: { data: null, loading: false, error: null }, reducers: {}, extraReducers: (builder) => { builder .addCase(fetchUser.pending, (state) => { state.loading = true }) .addCase(fetchUser.fulfilled, (state, action) => { state.loading = false state.data = action.payload }) .addCase(fetchUser.rejected, (state, action) => { state.loading = false state.error = action.error.message }) }, })
1
thunkAPI is powerful
The second argument to your payload creator gives you dispatch, getState, rejectWithValue() (for controlled errors), signal (for cancellation), and more.
2
Cancel with AbortController
createAsyncThunk exposes an AbortSignal via thunkAPI.signal. Pass it to fetch/axios to cancel in-flight requests when the component unmounts.
3
rejectWithValue for known errors
Throw inside the creator and RTK puts error info in action.error. But return thunkAPI.rejectWithValue(data) gives you full control — that data lands in action.payload in the rejected case.

Redux Middleware

Middleware sits between dispatch and the reducer. It can inspect, modify, delay, or cancel every action that passes through.

dispatch(action)
→
middleware 1
→
middleware 2
→
middleware N
→
reducer
The signature: Every middleware is a curried function: store => next => action => { ... }. Call next(action) to pass it forward. Don't call it to swallow the action.
// Middleware signature — three nested arrow functions const myMiddleware = (store) => (next) => (action) => { // Before reducer console.log('dispatching:', action) const result = next(action) // passes to next middleware / reducer // After reducer — state is already updated here console.log('new state:', store.getState()) return result }
// Register middleware in configureStore const store = configureStore({ reducer: rootReducer, middleware: (getDefaultMiddleware) => getDefaultMiddleware().concat(myMiddleware, analyticsMiddleware), // getDefaultMiddleware() already includes redux-thunk }) // ─── Real-world middleware examples ────────────────────────────────── // 1. Logger const logger = store => next => action => { console.group(action.type) console.info('dispatching', action) const result = next(action) console.log('next state', store.getState()) console.groupEnd() return result } // 2. Analytics — fire events on specific actions const analytics = store => next => action => { if (action.type === 'tickets/purchased') { trackEvent('ticket_purchase', { amount: action.payload.amount }) } return next(action) } // 3. Auth gate — block actions requiring auth const authGate = store => next => action => { if (action.meta?.requiresAuth && !store.getState().auth.isLoggedIn) { next({ type: 'auth/showLoginModal' }) return // swallow original action } return next(action) }
Built-in / common
redux-thunk included in RTK
Handles dispatched functions. Already in getDefaultMiddleware().
serializability check included
Warns if non-serializable values (Date, Map, function) enter the store.
redux-logger
Logs prev state, action, next state to DevTools console. Install separately.
Custom use cases

Queue actions when offline, replay when connection restores.

Redux Saga — complex async flows

Saga uses ES6 generators to manage side effects. Instead of callbacks or promise chains, you write async logic that reads like synchronous code. The middleware runs your generators, handles the async parts, and resumes them when ready.

Prerequisites: Understand ES6 generators (function*, yield) before Saga. A generator function pauses at each yield and resumes when .next() is called — Saga's middleware does this automatically.
// Generator refresher — Saga builds on this function* example() { const a = yield 'first pause' // pauses here, a = whatever .next(val) passes in const b = yield 'second pause' return a + b }
// Core effect creators — these are plain objects, not real calls import { call, put, take, takeEvery, takeLatest, select, all, fork, cancel } from 'redux-saga/effects' // call(fn, ...args) → run async fn, pause until it resolves // put(action) → dispatch an action to the store // take(actionType) → wait for a specific action // select(selector) → read from state // all([...effects]) → run effects in parallel (like Promise.all) // fork(saga) → start a non-blocking child saga // cancel(task) → cancel a forked task
// Worker saga — does the actual async work function* fetchUserSaga(action) { try { yield put({ type: 'users/loading' }) const user = yield call(api.getUser, action.payload.id) yield put({ type: 'users/success', payload: user }) } catch (err) { yield put({ type: 'users/error', payload: err.message }) } } // Watcher saga — listens for actions and forks workers function* watchFetchUser() { yield takeLatest('users/fetchRequested', fetchUserSaga) // takeLatest: if dispatched again before the first finishes, cancel the first // takeEvery: run all concurrently — no cancellation // takeLeading: ignore subsequent dispatches until current one finishes } // Root saga — combines all watchers export function* rootSaga() { yield all([ watchFetchUser(), watchCreateEvent(), watchPayment(), ]) }
// Advanced: race — take whichever resolves first, cancel the other import { race, delay } from 'redux-saga/effects' function* fetchWithTimeout(action) { const { response, timeout } = yield race({ response: call(api.getData, action.payload), timeout: delay(5000), }) if (timeout) { yield put({ type: 'data/timeout' }) } else { yield put({ type: 'data/success', payload: response }) } } // Channel: communicate between sagas or with external events import { eventChannel, END } from 'redux-saga' function createWebSocketChannel(socket) { return eventChannel(emit => { socket.on('message', emit) socket.on('close', () => emit(END)) return () => socket.close() // cleanup on unsubscribe }) }
Testing superpower: Because effects are plain objects, testing sagas requires no mocking. expect(generator.next().value).toEqual(call(api.getUser, 'id')) — you test the description of the effect, not the effect itself.

When to use what

The right tool depends on how complex your async flows are, not on team preference.

Redux Thunk
vs
Redux Saga
Use Thunk when...
Simple API calls
Fetch → success/error. This is 80% of all async needs. createAsyncThunk handles it in 10 lines.
Small team / new project
Thunk has almost no learning curve. Everyone understands async/await already.
Sequential async steps
Two or three API calls in a row, each depending on the last. Still readable with async/await.
You're using RTK Query
RTK Query handles caching, deduplication, and re-fetching. You barely need manual thunks.
Use Saga when...
Complex orchestration
Flows that involve watching for multiple actions, retries, timeouts, cancellation — these get messy fast with thunks.
Real-time / WebSockets
eventChannel makes WebSocket events act like Redux actions. Elegant, testable.
Race conditions matter
takeLatest, race(), and cancel() give you fine-grained control impossible with promise-based thunks.
You need thorough tests
Testing sagas without mocking any network calls is a genuine superpower for critical flows.
Modern reality: Most new projects use RTK + RTK Query and never need Saga. Saga shines in large existing codebases with complex flows, or in apps with heavy real-time requirements.
