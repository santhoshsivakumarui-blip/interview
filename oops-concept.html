pillar — encapsulation

Hide the state, expose only deliberate behavior

Internal fields stay private. Outside code can only change state through methods that enforce the rules.

-balance: numberprivate — unreachable directly
+deposit(amount) +withdraw(amount) +getBalance() 
class Account {
  #balance = 0; // truly private

  deposit(amount) {
    if (amount <= 0) throw new Error("invalid");
    this.#balance += amount;
  }
  getBalance() { return this.#balance; }
}
// account.#balance is a syntax error from outside the class

pillar — inheritance

Share behavior down a hierarchy

A subclass gets everything the parent has for free, then adds or changes only what's actually different.

Animaleat(), sleep()
Dog+ bark()
Cat+ meow()
class Animal {
  eat()   { return `${this.name} is eating`; }
  sleep() { return `${this.name} is asleep`; }
}
class Dog extends Animal {
  bark()  { return "Woof!"; }
}
// new Dog().eat() works — inherited, not rewritten

structure — classes & objects

The blueprint, and what gets built from it

A class defines shape and behavior. An object is one concrete instance of that shape, holding its own data.

Class

A template: field names, method definitions, no actual data of its own yet.

Object / Instance

A real value in memory built from the class, with its own copy of the instance fields.

Multiple instances

Every new Animal() call produces an independent object — changing one never touches another.

structure — constructors & this

Setting up a new instance

The constructor runs once, at creation — this always refers to the specific instance being built or called on.

class Dog {
  constructor(name) {
    this.name = name; // "this" = the instance being built
  }
  bark() { return `${this.name} says woof`; }
}
const dog1 = new Dog("Rex");
const dog2 = new Dog("Fido");

structure — static vs instance members

Belongs to the class, or to one object

Static members are shared across every instance. Instance members exist separately, once per object.

static (class-level)
Counter.total = 3
instance-level
c1.id = 1
c2.id = 2
c3.id = 3
class Counter {
  static total = 0;         // one copy, shared
  constructor() {
    this.id = ++Counter.total; // one per instance
  }
}
relationships — overriding vs overloading

Redefine a method, or offer several versions of it

class Animal { speak() { return "..."; } }
class Dog extends Animal {

}
// runtime picks the subclass's version — this is polymorphism in action
// JS has no true overloading — simulate with one function:
function area(a, b) {
  if (b === undefined) return a * a;       // square
  return a * b;                             // rectangle
}
// same name, behavior branches on arguments given
relationships — composition vs inheritance

"has-a" vs "is-a"

Inheritance models what something is. Composition models what it's made of — and is usually the more flexible default.

class Duck extends Bird {
  fly() { /* inherited flight logic */ }
}
// tightly coupled to Bird's shape — hard to reuse fly() elsewhere
class Duck {
  constructor() { this.flyBehavior = new FlyWithWings(); }
  fly() { return this.flyBehavior.fly(); }
}
// swap this.flyBehavior = new CannotFly() for a penguin — no rewrite
relationships — interface vs abstract class

A contract, or a partly-built base

interface Flyable { fly(): void; }
// pure contract — no shared code, no shared state
// a class can implement many interfaces at once
abstract class Bird {
  fly() { this.flap(); } // shared, concrete logic
  abstract flap(): void;  // left for subclasses
}
// carries real implementation — a class extends only one

Use an interface

You only need to guarantee a shape or capability, with no shared implementation.

Use an abstract class

Subclasses share real logic, and only a few specific steps genuinely vary.
