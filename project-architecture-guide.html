Project Architecture Guide

A practical checklist for building resilient Node.js services with retry logic, middleware, shared services, and reusable utilities.

1. Recommended project structure

A clean Node.js project usually separates routes, controllers, services, repositories, middlewares, and utilities so each layer has one job.

Routes / controllers

Handle HTTP requests, parse input, and delegate to services. Keep them thin and free of business rules.

Services

Encapsulate business logic and orchestrate repositories, caches, queues, and external clients.

Repositories / data access

Own database and persistence details so the rest of the app stays decoupled from storage mechanics.

Shared utilities

Provide common helpers for logging, validation, errors, pagination, dates, and serialization.

2. Retry Mechanisms

Retries are useful for transient failures, but they should be deliberate and bounded. A good retry strategy avoids amplifying outages.

Exponential backoff

Retry with increasing delays such as 100ms, 200ms, 400ms, and stop after a max attempt count.

Jitter

Add randomness to retry timing to avoid synchronized retries from many clients.

Idempotency

Ensure retried requests do not create duplicates by using unique request IDs or idempotency keys.

Retryable errors

Retry only network timeouts, 429, 5xx, and connection resets — not validation or authorization failures.

const delay = (ms) => new Promise(r => setTimeout(r, ms));

async function withRetry(fn, { retries = 3, baseDelay = 100, maxDelay = 1000 } = {}) {
  let attempt = 0;
  while (true) {
    try {
      return await fn();
    } catch (err) {
      if (attempt >= retries || !isRetryable(err)) throw err;
      const wait = Math.min(baseDelay * 2 ** attempt, maxDelay) + Math.floor(Math.random() * 50);
      await delay(wait);
      attempt += 1;
    }
  }
}

3. Common Middlewares in Node.js

Middleware keeps the application layered, observable, and easier to evolve.

Request logging

Log request ID, method, route, latency, and status for debugging and tracing.

Error handling

Catch thrown errors and format consistent API responses without leaking internals.

Authentication

Verify tokens, user roles, and permission scopes before business logic runs.

Validation

Validate body, params, and query payloads early with schemas like Joi or Zod.

Rate limiting

Protect APIs from abuse and accidental traffic spikes with per-user and per-IP limits.

Compression

Reduce payload size and improve response performance for JSON and text responses.

4. Common Services

Services encode reusable business logic and keep controllers thin.

Auth service

Handles token generation, verification, refresh logic, and role checks.

User service

Contains user profile, onboarding, and account-related workflows.

Notification service

Sends emails, SMS events, or push notifications through provider adapters.

Payment service

Encapsulates provider integrations and payment state transitions.

Cache service

Wraps Redis or in-memory cache operations for reads and cache invalidation.

Queue service

Publishes jobs to queues for asynchronous and background processing.

5. Common Utilities

Utilities improve consistency and reduce duplication across the project.

Http client

Centralizes outbound HTTP calls with timeout, retry, and logging behavior.

Logger utility

Provides structured logs with correlation IDs and environment-aware formatting.

Config loader

Reads environment variables and validates required settings at startup.

Error formatter

Converts domain errors into standard API responses and status codes.

Pagination helper

Builds consistent page, limit, and cursor-based pagination helpers.

Crypto utility

Wraps hashing, signing, token helpers, and secret handling in one place.

7. Practical checklist

Use structured logging with request IDs and correlation IDs.
Wrap external calls with timeouts, retries, and circuit-breaker-like safeguards.
Prefer dependency injection so services are easy to test and swap.
Keep configuration in one place and validate required env variables at startup.
Define consistent error shapes for API responses and background jobs.
