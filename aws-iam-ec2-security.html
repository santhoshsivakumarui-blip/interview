offset 000 — IAM building blocks

Users, groups, roles, policies

Four primitives. Nearly every IAM mistake comes from confusing which one to use.

Users

Individual identities for people. Avoid creating users for machine-to-machine access — reach for roles instead.

Groups

Attach policies to groups, not individual users, so permissions stay manageable as people join and leave.

Roles

Temporary credentials assumed by services, EC2 instances, or federated users. No long-lived secrets to leak.

Policies

JSON documents of Allow/Deny statements. Identity-based, resource-based, permission boundaries, and org-wide SCPs.

Session Manager

Shell access over the SSM agent — no inbound port, no bastion host, every session recorded to CloudTrail and optionally S3.

If SSH is unavoidable

Private subnet, bastion host or VPN only, key-based auth, root login disabled.

Instance role, not access keys

offset 004 — common failures

What actually gets accounts breached

✕

Access keys hardcoded in code or AMIs

Leaked in git history permanently, often scraped by bots within minutes of a public push.

FixUse IAM roles / instance profiles instead of long-lived keys.
✕

Wildcard policies to "get it working"

"Action":"*" on "Resource":"*" means one leaked credential is a full account compromise.

FixScope every policy down to exact actions and resource ARNs.
✕

SSH open to 0.0.0.0/0

A constant target for automated brute-force and credential-stuffing scans.

FixRestrict to known IPs, or drop the port and use Session Manager.
✕

No MFA on privileged accounts

Single-factor compromise of one password becomes full administrative access.

FixEnforce MFA via an IAM policy condition on sensitive actions.
✕

Public S3 buckets holding backups or secrets

One of the most common real-world breach vectors, usually from a default left unchanged.

FixBlock public access at the account level, verify with Access Analyzer.
offset 005 — hands-on order

Build it in this sequence

01

Lock the root account

Enable MFA, store credentials offline, stop using it day to day.

02

Create an IAM user + group with MFA

No console root usage from this point forward.

03

Launch EC2 with an IAM role attached

No access keys anywhere on the instance.

04

Lock the security group down

Only the ports and sources the app actually needs.

05

Enable Session Manager, disable SSH

Verify you can still get a shell with zero open inbound ports.

06

Enforce IMDSv2, encrypt the EBS volume

Close the SSRF credential-theft path, encrypt at rest.

07

Turn on CloudTrail, GuardDuty, Config

Now you can see what happens, not just prevent it.

08
